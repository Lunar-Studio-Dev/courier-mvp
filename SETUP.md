# TPC India Courier Management System — Local Setup Guide

## Prerequisites

| Tool       | Version  | Install                                      |
|------------|----------|----------------------------------------------|
| Node.js    | >= 18    | https://nodejs.org or `nvm install 18`       |
| pnpm       | 9.x      | `corepack enable && corepack prepare pnpm@9` |
| Docker     | any      | https://docs.docker.com/get-docker/          |
| jq         | any      | `sudo apt install jq` / `brew install jq`   |

## 1. Clone and Install

```bash
git clone <repo-url> courier
cd courier
pnpm install
```

## 2. Environment Configuration

The project uses an `env-manifest.json` file as the single source of truth for all environment variables. The setup script reads this manifest and creates independent `.env` files for each package/app.

### First-time setup (dev defaults)

```bash
pnpm setup
```

This creates `.env` files at:
- `apps/api/.env`
- `apps/web/.env`
- `packages/database/.env`
- `packages/services/.env`
- `packages/logger/.env`
- `.env` (root — used by dotenv-cli/turbo)

Each `.env` file also gets a matching `.env.example` committed to git.

### Database connection

The default `DATABASE_URL` is pre-configured to work with the included Docker Compose setup — no changes needed if you use `docker compose up -d`.

If you're using a different PostgreSQL instance, update `DATABASE_URL` in all `.env` files:

```
DATABASE_URL=postgresql://<user>:<password>@<host>:<port>/<database>
```

> The same `DATABASE_URL` is written to all packages that need it. If you change it in one place, run `pnpm setup:force` to regenerate all files, or manually update each `.env`.

### Other setup modes

```bash
# Interactive prompts for every variable (production)
pnpm setup:prod

# Backup existing .env files and regenerate from scratch
pnpm setup:force
```

### Validate environment

```bash
pnpm check:env
```

Reports any missing variables per package against the manifest.

## 3. Start the Database

The project includes a `docker-compose.yml` that runs PostgreSQL 15. Start it with:

```bash
docker compose up -d
```

This starts a PostgreSQL container with:
- **User:** `postgres`
- **Password:** `postgres`
- **Database:** `dev`
- **Port:** `5432`

Data is persisted in a Docker volume (`pg_data`), so it survives container restarts.

The default `DATABASE_URL` generated by `pnpm setup` already matches this configuration:

```
DATABASE_URL=postgresql://postgres:postgres@localhost:5432/dev
```

### Useful Docker commands

```bash
# Stop the database
docker compose down

# Stop and delete all data
docker compose down -v

# View logs
docker compose logs -f postgresdb

# Open a psql shell
docker exec -it postgresdb psql -U postgres -d dev
```

## 4. Run Migrations

```bash
pnpm db:migrate
```

This applies all Drizzle ORM migrations from `packages/database/drizzle/`.

If you need to generate new migration files after schema changes:

```bash
pnpm db:generate
```

## 5. Seed Data

### Master data (required)

Seed the core lookup tables — admin user, destinations, pricing rules, and invoices:

```bash
pnpm db:seed:all
```

This runs the following in order:
1. `db:seed` — base seed (admin user, core config)
2. `db:seed:destinations` — Indian cities/pincodes
3. `db:seed:admin` — admin account
4. `db:seed:pricing` — pricing rules and slabs
5. `db:seed:invoices` — sample invoices

### Demo data (optional, recommended for presentations)

Populate the system with realistic demo branches, customers, and shipments:

```bash
pnpm seed:demo
```

This runs:
1. `db:seed:demo-branches` — 6 branches across Indian cities
2. `db:seed:demo-customers` — 12 customers with addresses and ID proofs
3. `db:seed:demo-shipments` — 150 shipments across all statuses with tracking history

All demo seeds are idempotent — safe to run multiple times.

### One-command database setup

To run migrations + all master seeds in one go:

```bash
pnpm db:setup
```

## Notifications (MVP)

The notification system uses a mock adapter by default. When no real SMS/email provider is configured, notifications are logged to the database and surfaced as toast messages in the admin dashboard. No external service setup is needed for development.

## Theming

The app supports light and dark modes via a toggle in the navigation bar (all three layouts: public site, admin dashboard, customer portal). The theme uses OKLCH color palettes defined in `apps/web/app/globals.css`. The default theme is light with system preference detection enabled.

## Troubleshooting

**`DATABASE_URL` is undefined**
Each app's `.env` must contain all variables needed by its dependency tree. Run `pnpm setup:force` to regenerate all `.env` files, then verify with `pnpm check:env`.

**Migrations fail**
Ensure the database container is running (`docker compose up -d`) and `DATABASE_URL` in `packages/database/.env` matches the Docker credentials. Check container status with `docker compose ps`.

**Seed scripts fail with duplicate errors**
Master seeds may not be idempotent. Reset the database by running `docker compose down -v && docker compose up -d`, then run `pnpm db:setup` followed by `pnpm seed:demo`.

**Port conflicts**
API defaults to port 8000, Web to port 3000. Change `PORT` in `apps/api/.env` or pass `--port` to Next.js if needed.
